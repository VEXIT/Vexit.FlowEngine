# Vexit Flow Engine Design Documentation (2026)

This document outlines the architecture for a lightweight, resumable, and parallel-capable workflow engine designed in C#.

## 1. Core Philosophy
- **Blueprint vs. Runtime**: Separate the "Recipe" (**Flow**) from the "Live Execution" (**Instance**).
- **Composite Design**: Use a tree structure (Physical Containers) to allow nesting of Groups, Steps, and Decisions.
- **Delegate Mapping**: Map JSON keys to C# methods using a `Dictionary<string, Delegate>` for high performance and clean code.

---

## 2. The Blueprint (Flow Definition)
This is the static JSON file that defines the server setup process.

### **Flow**
The top-level container for the entire process.
- `Id`: Unique identifier for the flow.
- `Root`: The starting `Activity` (usually a `Group`).

### **Activity (Abstract Base)**
The base class for everything that happens in the flow.
- `Id`: Unique ID (e.g., "setup-nginx").
- `Name`: Friendly display name.

### **Step : Activity**
A leaf node that executes actual code.
- `HandlerKey`: String key mapped to a C# method (e.g., "InstallPostgres").
- `Inputs`: `Dictionary<string, string>` used to pass parameters (e.g., `"Version": "15"`).

### **Group : Activity**
A container for other activities.
- `Children`: `List<Activity>` (Physical nesting).
- `RunInParallel`: Boolean. If true, the engine uses `Task.WhenAll`.

### **Decision : Activity**
A branching node.
- `Expression`: String logic (e.g., `"OS == 'Ubuntu'"`).
- `TruePath`: `Activity` to execute if true.
- `FalsePath`: `Activity` to execute if false.

---

## 3. The Runtime State (The Instance)
This is the state object saved to a JSON file on the server to track progress.

### **FlowInstance**
The live snapshot of a running flow.
- `FlowId`: Reference to the Blueprint.
- `Data`: `Dictionary<string, object>` (Global variables like `ServerIP`, `AdminPass`).
- `ActivityInstances`: A flattened list of the state of every activity.

### **ActivityInstance**
- `ActivityId`: Points back to the Activity in the Blueprint.
- `IsCompleted`: Boolean flag for resumption.
- `Data`: Local output/logs from that specific step.
- `Error`: Stores exception messages if the step fails.

---

## 4. Execution Engine Logic

### **The Delegate Registry**
Instead of one class per step, group related logic into Domain Services (e.g., `ServerSetupService`) and map them via delegates.

```csharp
public delegate Task StepFunc(Dictionary<string, string> inputs, FlowInstance instance);

// Registry Mapping
var registry = new Dictionary<string, StepFunc> {
    { "InstallPostgres", _service.InstallPostgres },
    { "SetupNginx",      _service.ConfigureNginx }
};
```

### The Execution Loop

The engine traverses the tree. If it hits a Step, it looks up the HandlerKey in the registry and awaits the function. If it hits a Group, it either loops or runs them in parallel.


## 5. Key Features for Provisioning

1. **Resumability**: Before executing any Step, the engine checks ActivityInstance.IsCompleted. If true, it skips to the next one. This allows the vxs command to resume after a crash.
2. **Inputs/Outputs**: Steps read from Step.Inputs and write results into FlowInstance.Data.
3. **Thread Safety**: When running Parallel Groups, the engine should ideally instantiate a new scoped Service to avoid shared-resource conflicts (like SSH sessions).
4. **Logic Evaluation**: Use a library like NCalc or Dynamic LINQ to evaluate Decision.Expression strings against the FlowInstance.Data dictionary at runtime.

## Inputs Outputs Architecture

### 1. The Concept
In a provisioning workflow, steps often need data generated by previous steps. Instead of hardcoding dependencies, we use **Mapping**.
- **Inputs**: Pull data from `FlowInstance.Data` (Global State) into a Step.
- **Outputs**: Push results from a Step back into `FlowInstance.Data`.


### 2. Step Model Definition


```csharp
public class Step : Activity
{
    public string HandlerKey { get; set; }
    
    /// <summary>
    /// Key = Parameter name in C# method.
    /// Value = Key name in FlowInstance.Data (or a static string).
    /// </summary>
    public Dictionary<string, string> Inputs { get; set; } = new();

    /// <summary>
    /// Key = Key name in the Dictionary returned by the C# method.
    /// Value = Key name where it should be saved in FlowInstance.Data.
    /// </summary>
    public Dictionary<string, string> Outputs { get; set; } = new();
}
```
### Execution Delegate


```csharp
/// <summary>
/// Steps now return a Dictionary of results for the engine to map.
/// </summary>
public delegate Task<Dictionary<string, object>> StepFunc(
    Dictionary<string, string> inputs, 
    FlowInstance instance);
```

### 3. Real-World Example: Server Provisioning


### 4. Engine Orchestration Logic

When the Vexit.FlowEngine processes a Step, it follows this internal sequence:

1. **Resolve Inputs**:
The engine looks at `Step.Inputs`. For each entry, it finds the corresponding value in `FlowInstance.Data` and prepares a local dictionary for the method.

2. **Execute**:
```csharp
var results = await handler(resolvedInputs, instance);
```

3. **Apply Outputs**:
The engine iterates through `Step.Outputs`. It takes the values from the `results` dictionary and writes them into the `FlowInstance.Data` global store.

### 5. Benefits for Vexit CLI

1. **Resumability**: Because outputs like `System_OS_Name` are saved in the `FlowInstance` JSON file, if you resume a setup (as seen in your log: "Do you wish to continue unfinished server setup?"), the engine doesn't need to re-run the detection steps.
2. **Validation**: Steps can output validation flags. A `Decision` node can then check if `System_RAM_MB > 2048` before deciding whether to install a heavy application.
3. **Final Reporting**: The success message at the end ("Access your server at: http://72.62.151.6") is populated by pulling the `ServerIP` from the `FlowInstance.Data` that was output by the earlier connection step.

## Sample Code
### Domain Model

```csharp
using System;
using System.Collections.Generic;
using System.Threading.Tasks;

namespace Vexit.FlowEngine.Models;
{
    #region --- BLUEPRINT (Definitions) ---

    /// <summary>
    /// Represents the top-level definition of a workflow process.
    /// This is the static "recipe" typically loaded from a JSON file.
    /// </summary>
    public class Flow
    {
        /// <summary>Unique identifier for the flow definition.</summary>
        public string Id { get; set; }

        /// <summary>The display name of the flow (e.g., "Full Server Provisioning").</summary>
        public string Name { get; set; }

        /// <summary>The version of this flow definition for schema migration purposes.</summary>
        public string Version { get; set; } = "2026.1";

        /// <summary>
        /// The starting point of the flow. Due to the composite design, 
        /// this can be a single Step, a Group, or a Decision.
        /// </summary>
        public Activity Root { get; set; }
    }

    /// <summary>
    /// The abstract base class for all components within a Flow.
    /// </summary>
    public abstract class Activity
    {
        /// <summary>Unique ID within the specific Flow (e.g., "install-pg-12").</summary>
        public string Id { get; set; }

        /// <summary>Friendly name of the activity for logging and UI.</summary>
        public string Name { get; set; }
    }

    /// <summary>
    /// Represents a leaf-node activity that executes a specific piece of code.
    /// </summary>
    public class Step : Activity
    {
        /// <summary>
        /// The key used to look up the C# delegate/method in the Handler Registry.
        /// </summary>
        public string HandlerKey { get; set; }

        /// <summary>
        /// Key = Parameter name in C# method.
        /// Value = Key name in FlowInstance.Data (or a static string).
        /// </summary>
        public Dictionary<string, string> Inputs { get; set; } = new();
    
        /// <summary>
        /// Key = Key name in the Dictionary returned by the C# method.
        /// Value = Key name where it should be saved in FlowInstance.Data.
        /// </summary>
        public Dictionary<string, string> Outputs { get; set; } = new();
    }

    /// <summary>
    /// A container activity that holds a collection of child activities.
    /// Supports both sequential and parallel execution.
    /// </summary>
    public class Group : Activity
    {
        /// <summary>The physical list of child activities (Steps, Decisions, or other Groups).</summary>
        public List<Activity> Children { get; set; } = new();

        /// <summary>If true, children are executed simultaneously via Task.WhenAll.</summary>
        public bool RunInParallel { get; set; }
    }

    /// <summary>
    /// A branching activity that evaluates logic to determine the next path.
    /// </summary>
    public class Decision : Activity
    {
        /// <summary>
        /// A string expression evaluated against the FlowInstance.Data 
        /// (e.g., "AppServer == 'NodeJs'").
        /// </summary>
        public string Expression { get; set; }

        /// <summary>The activity to execute if the expression evaluates to true.</summary>
        public Activity TruePath { get; set; }

        /// <summary>The activity to execute if the expression evaluates to false.</summary>
        public Activity FalsePath { get; set; }
    }

    #endregion

    #region --- RUNTIME (State & Instance) ---

    /// <summary>
    /// Defines the delegate signature for step execution.
    /// </summary>
    /// <param name="inputs">The inputs defined in the Step Blueprint.</param>
    /// <param name="instance">The live state of the current flow.</param>
    public delegate Task StepFunc(Dictionary<string, string> inputs, FlowInstance instance);

    /// <summary>
    /// Represents the live execution state of a Flow.
    /// This object is what you serialize to JSON for persistence and resumption.
    /// </summary>
    public class FlowInstance
    {
        /// <summary>Unique ID for this specific execution run.</summary>
        public Guid Id { get; set; } = Guid.NewGuid();

        /// <summary>The ID of the Flow definition being executed.</summary>
        public string FlowId { get; set; }

        /// <summary>Global variables and shared state data for the entire flow.</summary>
        public Dictionary<string, object> Data { get; set; } = new();

        /// <summary>
        /// A collection of states for every activity that has been processed.
        /// Useful for resuming and auditing.
        /// </summary>
        public List<ActivityInstance> ActivityInstances { get; set; } = new();

        /// <summary>The current high-level status (e.g., Running, Completed, Faulted).</summary>
        public string Status { get; set; } = "Running";
    }

    /// <summary>
    /// Represents the execution state of an individual activity within a FlowInstance.
    /// </summary>
    public class ActivityInstance
    {
        /// <summary>Points back to the Activity.Id in the Blueprint.</summary>
        public string ActivityId { get; set; }

        /// <summary>Indicates if this specific activity finished successfully.</summary>
        public bool IsCompleted { get; set; }

        /// <summary>Stores the error message if the activity failed.</summary>
        public string Error { get; set; }

        /// <summary>Execution-specific data or logs for this individual step.</summary>
        public Dictionary<string, string> Data { get; set; } = new();

        /// <summary>Timestamp of when the activity started.</summary>
        public DateTime StartedAt { get; set; } = DateTime.UtcNow;

        /// <summary>Timestamp of when the activity finished.</summary>
        public DateTime? CompletedAt { get; set; }
    }

    #endregion
}
```

### Consumer CLI App - Server Provisioning App


As an example of a consumer CLI app, we can use the VxServerCli project.


#### Summary of the Layering

- **Vexit.FlowEngine (Core Lib)**: Contains the models (Flow, Step, etc.) and the StepFunc delegate definition.
- **ServerSetupService (CLI Service)**: Contains the actual provisioning logic. This stays in your CLI project because it's specific to "Server" work.
- **FlowHandlerRegistry (CLI Bridge)**: A simple dictionary that links the two. 
- By using StepFunc as a delegate, you avoid creating dozens of tiny classes for each step while still benefiting from Dependency Injection for logging and configuration.

#### Flow Handler Registry

```csharp
using Vexit.FlowEngine.Models;

namespace Vexit.VxServerCli;

public class FlowHandlerRegistry 
{
    private readonly Dictionary<string, StepFunc> _handlers;

    public FlowHandlerRegistry(ServerSetupService service)
    {
        // Map the keys used in your JSON to the C# methods
        _handlers = new Dictionary<string, StepFunc>(StringComparer.OrdinalIgnoreCase)
        {
            { "InstallPostgres", service.InstallPostgres },
            { "SetupNginx",      service.ConfigureNginx }
        };
    }

    public StepFunc GetHandler(string key) => 
        _handlers.TryGetValue(key, out var handler) ? handler : null;
}
```

#### Program.cs

```csharp
// Program.cs
using Microsoft.Extensions.DependencyInjection;
using Microsoft.Extensions.Hosting;
using Vexit.FlowEngine.Models;

var builder = Host.CreateApplicationBuilder(args);

// Register the logic service
builder.Services.AddScoped<ServerSetupService>();

// Register the registry as a singleton or scoped
builder.Services.AddSingleton<FlowHandlerRegistry>();

// Register the Engine itself
builder.Services.AddScoped<FlowEngine>();

using var host = builder.Build();

// --- EXAMPLE EXECUTION IN A COMMAND ---
var engine = host.Services.GetRequiredService<FlowEngine>();
var registry = host.Services.GetRequiredService<FlowHandlerRegistry>();

// Load your blueprint (e.g., from a JSON file)
var blueprint = new Flow { /* ... */ }; 
var instance = new FlowInstance { FlowId = blueprint.Id };

// Execute a step
var step = new Step { HandlerKey = "InstallPostgres", Inputs = new() { ["Version"] = "16" } };
var handler = registry.GetHandler(step.HandlerKey);

if (handler != null) 
{
    await handler(step.Inputs, instance);
}
```
